# 用豆包端到端实时语音大模型打造家庭语音助手（火山引擎 Realtime API）

## 前言

上一篇用纯开源方案搭了个语音助手，能用但体验一般——延迟高、唤醒不灵、识别不准。这篇换一个思路：直接用火山引擎（字节跳动）的端到端实时语音大模型，体验直接拉满。

## 什么是"端到端实时语音"

传统语音助手的链路是分步的：

```
录音 → STT(语音转文字) → LLM(思考) → TTS(文字转语音) → 播放
```

每一步都有延迟，加起来就很慢。

端到端实时语音是**一个模型搞定所有事**：

```
麦克风音频流 ←→ 端到端大模型 ←→ 音响音频流
```

音频进、音频出，中间不需要你管 STT/LLM/TTS。模型自己听、自己想、自己说。延迟极低，接近真人对话。

## 架构（极简）

```
USB麦克风 → 服务器 → WebSocket → 火山引擎端到端模型 → WebSocket → 服务器 → 音响
              ↑                                                    ↓
         实时推送音频(20ms/包)                              实时接收音频并播放
```

服务器只做两件事：
1. 从麦克风读音频，推给火山
2. 从火山收音频，推给音响

所有"智能"都在云端模型里。

## 硬件（同上一篇）

- Linux 服务器（任何能跑 Python 的机器）
- USB 无线麦克风
- 3.5mm 有线音响

## 技术实现

### 核心依赖

```
websockets    # WebSocket 客户端
sounddevice   # 麦克风录音
numpy         # 音频处理
```

### 协议说明

火山引擎 Realtime API 使用**自定义二进制协议**通过 WebSocket 传输：

- 4 字节 header（协议版本、消息类型、序列化方式）
- 可选字段（event ID、session ID）
- payload（JSON 或原始音频二进制）

### 核心流程

```python
async def run():
    # 1. WebSocket 连接
    async with websockets.connect(WS_URL, additional_headers=headers) as ws:
        # 2. StartConnection
        await ws.send(build_client_event(1))
        
        # 3. StartSession（配置方言、人设）
        await ws.send(build_client_event(100, session_id, session_payload))
        
        # 4. 并发：发送音频 + 接收事件
        await asyncio.gather(
            send_audio(),      # 麦克风 → 火山
            receive_events(),  # 火山 → 音响
        )
```

### 音频发送（20ms 一包）

```python
async def send_audio():
    with sd.InputStream(samplerate=48000, channels=1, dtype="int16") as stream:
        while running:
            chunk, _ = stream.read(960)  # 20ms @ 48kHz
            chunk_16k = chunk[::3]       # 降采样到 16kHz
            frame = build_audio_frame(chunk_16k.tobytes(), session_id)
            await ws.send(frame)
            await asyncio.sleep(0.02)
```

### 音频接收与播放

```python
async def receive_events():
    while running:
        data = await ws.recv()
        parsed = parse_server_frame(data)
        
        if parsed["event_id"] == 352:  # TTSResponse - 音频数据
            aplay_proc.stdin.write(parsed["payload"])  # 流式写入 aplay
        elif parsed["event_id"] == 450:  # ASRInfo - 检测到说话
            aplay_proc.kill()  # 打断播放
```

## 特性

| 特性 | 说明 |
|------|------|
| 无需唤醒词 | 服务端 VAD 自动检测你在说话 |
| 实时对话 | 延迟极低，接近真人 |
| 方言支持 | 陕西话、四川话、东北话 |
| 自定义人设 | bot_name + system_role + speaking_style |
| 打断支持 | 你说话时模型自动停止回复 |
| 上下文记忆 | 支持 dialog_id 保持对话连续性 |
| 内置联网 | 可开启搜索能力 |

## 计费

- 按 tokens 计费，不按连接时长
- 挂着不说话不花钱
- 免费额度：1,000,000 tokens（约 1000 轮对话）
- 到期时间：2026-12-01

## 与方案一的对比

| | 方案一（纯开源） | 方案二（豆包端到端） |
|--|--|--|
| 延迟 | 4-8 秒 | <1 秒 |
| 识别准确率 | 一般（Whisper small） | 很高（Seed-ASR） |
| 方言支持 | 差 | 好（陕西/四川/东北） |
| 唤醒 | 需要（体验差） | 不需要 |
| 打断 | 不支持 | 支持 |
| LLM | 自定义（Hermes/DeepSeek） | 字节豆包模型 |
| 隐私 | 数据不出局域网 | 音频上传到字节云端 |
| 成本 | 完全免费 | 有免费额度，超出付费 |
| 复杂度 | 多组件拼装 | 一个 WebSocket 搞定 |

- 追求极致体验的家庭语音助手

- 需要方言支持的场景

## 踩坑记录

1. **麦克风采样率**：USB 无线麦只支持 48kHz，需降采样到 16kHz 再发送
2. **播放采样率**：声卡不支持 24kHz 直接播放，用 aplay 的 plughw 设备自动转换
3. **回声问题**：音响播放时麦克风会录到回声，导致模型"自己跟自己聊"。解决方案是检测到 ASRInfo 事件时停止播放
4. **二进制协议**：火山的 WebSocket 不是标准 JSON，是自定义二进制帧，需要手动组装/解析

## 总结

如果你只是想要一个"能用且体验好"的语音助手，豆包端到端方案是目前最省事的选择。一个 Python 文件、一个 WebSocket 连接，就能实现接近商业产品的语音对话体验。

代价是 LLM 不可控（用的是字节的模型）、数据要上云。如果你对这两点敏感，方案一（纯开源）更适合你。

两套方案可以共存——日常用端到端方案享受体验，需要隐私或自定义 LLM 时切回开源方案。
